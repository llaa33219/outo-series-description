# agnetouto
## 개발 이유와 현재의 목적
처음에는 codebuff, opencode와 같은 멀티에이전트 기반 코딩 툴을 제작하기 위하여 설계됨.  
하지만 현재는 코딩 에이전트라는 목적이 흐려지고 전반적인 모든 에이전트의 기반이 되는 라이브러리라는 목적으로 개발되고 있음.
## 특징 정리
1. **완전한 평등**: 유저와 에이전트, 에이전트와 에이전트 모두 다 동등하게 취급하며, 제한이 걸리지 않음. 동일한 형식으로 통신.
2. **간단한 사용**: 선언할 것이 많지 않고, 간결한 문법으로 쉽게 사용할 수 있음
3. **적절한 문서화**: 여러 기능이 문서화가 잘 되어 있어, ai 에이전트가 이 라이브러리를 쉽게 이해할 수 있음.
4. **다양한 공급자 지원**: 여러가지 공급자를 지원하여 공급자 지원 문제로 인하여 개발이 불가하게 되는 상황이 일어나지 않음.
5. **다양한 사용 상황**: openclaw, claude code 등과 같은 에이전트들을 이 라이브러리 기반으로 제작할 수 있으며, 도구와 에이전트 역할 정의를 이용하여 랭체인/n8n 등에서 할 수 있는 자동화 파이프라인을 모방할 수 있음.(단, 원칙에 따라서 강제적인 파이프라인 제작에는 제한이 있음)
## 대체 어떤 상황에서 사용해야 하는가
1. **간단하게 에이전트를 구축**해야 하는 상황
2. 강제적인 파이프라인 보다는 **에이전트가 유동적으로 판단하는 파이프라인**이 필요한 경우
3. 특정 에이전트를 개조하려 시도했지만 실패하여, **자체적으로 에이전트 시스템을 구축**해야 하는 상황
# outobot
## 개발 이유와 현재의 목적
openclaw가 기능이 너무 과도하고 설정이 어려우며, 공급자가 잘못 연동되는 경우가 생기는 바람에 "이럴바에 직접 만들지"라는 생각이 들어 제작됨.  
현재도 같은 목적으로 개발되고 있으며, 메모리 기능, 에이전트 자가복제 기능(다른 컴퓨터 등에 자기 설정을 직접 복제), 스킬의 적극적인 이용 등 다양한 새로운 기술과 기능을 고려하여 기능 추가를 진행하고 있음.
## 특징 정리
1. **간단한 설정**: 설치 시 바로 localhost:7227로 접속하여 gui로 간단하게 설정할 수 있음
2. **명확한 제거**: uninstall.sh로 매우 정확하고 깔끔하게 제거하여 "내가 잘못한 것인가 제거가 잘못된 것인가"라는 문제가 발생하는 것을 차단함
3. **멀티에이전트 친화적 설계**: 다양한 에이전트가 협업하는 것이 기본 기능으로 설계되어 다양한 에이전트의 협업을 쉽게 경험할 수 있음
4. **강력한 커스텀**: me.md 파일에 에이전트에 대한 것을 서술하면, 그 파일이 시스템 프롬프트에 주입되어, 내용을 더욱 정확하게 복종하게 됨.
5. **확장 가능성과 개발 편의성**: 직관적이고 잘 나누어져 설계되어 에이전트를 개조하거나, 필요한 부분을 변경하는 것이 유리하고 신기술을 더욱 빨리 적용할 수 있음.
6. **통합과 격리**: 에이전트는 distrobox를 이용하여 생성한 아치리눅스 컨테이너 안에서 작업함으로써 호스트와의 적절한 통합을 유지하며, 환경을 컨테이너 환경에서 작업할 수 있음
## 대체 어떤 상황에서 사용해야 하는가
1. 에이전트를 **빠르게 설치**하고 다른 컴퓨터에서도 **같은 환경을 구성**하고 싶은 상황
2. **에이전트의 설치가 너무 어려워**서 도저히 설치를 할 수 없는 사람
# outowiki
## 개발 이유와 현재의 목적
**실제 위키는 문서가 많아져도 관리/검색이 쉽게 가능하다**라는 철학 하에 개발되고 있으며,  
기존 llm wiki, md문서 관리 툴 들과는 다르게 **실제 위키의 가이드를 적극 수용**하여 개발되고 있음.  
기본적으로 문서가 5000, 500000, 500000개가 되는 상황을 생각하여 제작되어 **실제 위키의 규칙을 따라** 적절한 탐색과 깔끔한 관리가 되게 하고 있음.  
## 특징 정리
1. **대용량 문서 관리 특화**: 문서가 500개만 넘어가도 관리를 버거워하는 기존 llm wiki들과는 다르게, 문서가 5000, 50만개가 되도 완벽하게 관리가 되도록 개발되고 있음.
2. **실제 위키 규칙을 수용**: 실제 위키의 규칙을 수용하여, 대규모 문서 관리의 기초 틀을 단단하게 잡고 있음
3. **직관적인 저장**: 위키의 분류 시스템을 폴더 기반으로 도입하여, 이 라이브러리를 이용하지 않는 에이전트가 폴더를 열람하더라도, 간단한 규칙만 알고 있다면 문서를 찾을 수 있으며, 마크다운 문서와 폴더로 관리되니 복사/붙여넣기로 문서를 이전시키거나 어딘가에 백업하기 매우 편리함
4. **내장 에이전트**: 라이브러리 자체의 내장 에이전트가 존재하여, 문서 저장과 검색을 진행함. 이 라이브러리를 도구로 사용하는 에이전트는 토큰을 낭비하지 않고, 라이브러리에 의존하여 쉽게 문서를 찾고 저장할 수 있음.
5. **변화 친화적**: 백터DB와 그래프DB를 쓰는 RAG기반의 메모리는 모델이 바뀜에 따라 벡터 차원이 일치하지 않게 되면 작동을 하지 않기 때문에, 신규 모델을 도입하기 쉽지 않음. 하지만 이 라이브러리는 llm 모델만을 사용하여 관리를 진행하여, 신규 모델이 출시되면 바로 도입할 수 있음.
6. **간단한 사용**: 간단한 선언 하나만으로 동작을 시작하며, 간단한 코드만으로 바로 llm wiki를 구축할 수 있음
## 대체 어떤 상황에서 사용해야 하는가
1. **llm wiki를 쉽고 간단하게 구축**하고 싶은 상황
2. **24/7에이전트의 메모리** 관리 등의 이유로 **대규모의 문서가 생성되고 관리**되어야 하는 상황
3. **신규 기술을 빠르게 받아들여** 메모리 시스템의 ai **모델을 업그레이드 하는 것이 중요한** 상황
4. 구축된 **wiki를 여기저기 옮기고 백업**해야 하는 상황
# outo-agentcore
## 개발 이유와 현재의 목적
outobot과 같은 에이전트가 생각보다 너무 거대해져서 문제가 발생해서 제작됨.  
에이전트에게 꼭 필요한 코어 부분만 넣어 다른 에이전트 시스템의 근간이 되게 하는 것이 목적임.
## 특징 정리
1. **최소한의 기능**: 에이전트가 돌아가기에 필요한 최소한의 기능을 탑제하고 있으며, 쓸대없는 기능을 더하여 확장하려 하지 않음
2. **깔끔한 통합**: cli 기반으로 사용이 가능하고, cli 명령어 또는 간단한 파일 수정/생성으로 에이전트 정의와 공급자 설정을 할 수 있어, 자동화된 에이전트 생성이나 다른 툴에 통합을 쉽게 진행할 수 있음
3. **쉬운 에이전트 정의**: 에이전트를 간단하고 쉽게 정의할 수 있어, 에이전트를 다양하게 정의해야하는 상황에 대응할 수 있음.
4. **명확한 제거**: ~/.outoac/ 와 이 라이브러리(파이썬 라이브러리 형식으로 배포되기 때문)를 제거하면, 모든 설정이 사라지며 완전히 이 에이전트를 제거할 수 있음.
## 대체 어떤 상황에서 사용해야 하는가
1. **agnetouto보다 더 쉽게 완성된 에이전트 시스템을 구축**하고 싶은 경우
2. **다양한 환경에 완벽히 동일한 에이전트 구성을 배포**하고 싶은 경우
3. **gui가 없는 환경**에서 **간단하게 에이전트를 설치하고 사용**하고 싶은 경우
4. 복잡한 에이전트를 개발하기에는 **기술이 부족**하거나 **이미 완성된 기반을 사용**하고 싶은 경우
